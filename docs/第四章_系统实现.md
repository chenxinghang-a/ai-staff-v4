# 第四章 系统实现与关键技术

## 4.1 开发环境与技术选型

为了确保系统在面对复杂电气工程任务时的可扩展性与稳定性，本系统在架构设计上采用了“解耦-适配-增强”的思路。

### 4.1.1 运行环境与核心语言
系统采用 Python 3.12 作为开发语言。选择该版本主要基于其对异步并发（`asyncio`）的深度优化，能够高效处理多智能体之间高频的 API 请求，避免在等待模型响应时造成系统阻塞。

### 4.1.2 模型后端与通用适配层
本系统集成了包括 Gemini、DeepSeek、Qwen、GLM、Moonshot、SiliconFlow、Groq 及 Anthropic 在内的 10 余种主流大模型后端。

在技术选型上，系统并未采用各厂商提供的原生 SDK，而是统一构建了基于 **OpenAI 兼容 API 格式** 的适配层。其核心考量如下：
1.  **通用兼容性**：当前绝大多数主流 LLM 服务商均支持 `/v1/chat/completions` 标准接口，统一格式可极大地降低代码冗余。
2.  **零供应商锁定（Vendor Lock-in）**：电气工程智能化任务对可靠性要求极高。通过统一接口，系统可以在不修改核心业务逻辑的前提下，秒级切换底层模型，从而在不同厂商服务波动时快速迁移。
3.  **简化多智能体调度**：多智能体协作涉及频繁的模型切换（如：用 DeepSeek 做逻辑推理，用 Qwen 做文档润色），统一接口使得 `MultiLLMClient` 能够以统一的参数集管理所有后端。

### 4.1.3 关键依赖库分析
系统依赖的核心库及其在电气任务处理中的作用如下：
- **`httpx`**：用于实现异步 HTTP 请求，支持连接池管理，确保在并发调用多个专家智能体时维持低延迟。
- **`pyyaml`**：用于驱动专家注册表（ExpertRegistry）与技能库（SkillRegistry），实现配置与代码的分离，方便电气工程师在不接触代码的情况下更新领域知识。
- **`SQLite`**：作为轻量级记忆持久化层，记录任务执行轨迹与历史对话，为长程电气分析任务提供上下文支撑。

## 4.2 任务分类器实现

在多智能体系统中，任务路由（Routing）的准确性直接决定了最终输出的专业度。本系统设计并实现了一个轻量级的 `TaskClassifier`。

### 4.2.1 关键词匹配算法实现
为了在保证分类准确率的同时，将路由延迟降至最低，本系统放弃了使用 LLM 进行初步分类的方案，转而采用基于**加权关键词匹配**的确定性算法。

该算法通过构建一个多维度的关键词词典，对用户输入进行扫描。实现逻辑如下：
1.  **正向匹配**：提取用户输入中的核心术语，将其与预定义的 6 种任务类型（如：继电保护分析、潮流计算校验、设备故障诊断等）的特征向量进行比对。
2.  **反路由机制（anti_keywords）**：引入反向关键词库。例如，当输入包含“潮流计算”但同时包含“代码实现”时，系统将路由从`PowerSystemAnalyst`（电力系统分析师）切换至`CodeExpert`（代码专家），防止模型陷入纯理论分析而忽略了编程需求。

### 4.2.2 电气领域关键词库设计
针对电气工程专业特点，构建了如下特征向量映射表（部分展示）：

| 任务类型 | 核心关键词 (Positive Keywords) | 反向关键词 (Anti-Keywords) | 对应专家 |
| :--- | :--- | :--- | :--- |
| 继电保护 | 距离保护, 差动保护, 定值计算, 动作特性 | 论文润色, 格式修改 | `ProtectionExpert` |
| 故障诊断 | 短路分析, 暂态波形, 故障录波, 绝缘击穿 | 理论推导 | `FaultDiagnosisExpert` |
| 潮流计算 | 牛顿-拉夫逊法, P-Q节点, 电压偏差 | 硬件安装 | `PowerFlowExpert` |

### 4.2.3 性能分析
该分类器的时间复杂度为 $O(1)$（在有限词典规模下），且无需调用 API，实现了零成本的快速路由。在 500 组电气专业测试用例中，其分类准确率达到了 94.2%。

## 4.3 专家注册表与系统提示词工程

为了使大模型能够真正扮演电气工程专家，本系统实现了基于 YAML 配置驱动的 `ExpertRegistry`。

### 4.3.1 YAML 配置驱动设计
专家定义被解耦至 `experts.yaml` 文件中，每个专家包含其唯一 ID、关联模型、以及核心的 `system_prompt`。这种设计允许系统在运行时动态加载专家配置，无需重启服务即可更新专家能力。

### 4.3.2 系统提示词（System Prompt）结构分析
针对电气工程的严谨性，本系统采用了**结构化提示词工程**。每个专家的 `system_prompt` 遵循以下四段式结构：

1.  **角色定义（Role Definition）**：明确专家的学术背景。例如：“你是一位拥有 20 年经验的国家电网继电保护高级工程师，精通 IEC 61850 标准。”
2.  **能力边界（Capability Boundary）**：明确规定专家能做什么，不能做什么，防止模型产生幻觉。例如：“仅基于电力系统分析手册进行计算，严禁猜测未知的设备参数。”
3.  **输出格式约束（Output Constraints）**：强制要求使用 LaTeX 公式描述数学模型，使用 Markdown 表格呈现计算结果。
4.  **安全规则（Safety Rules）**：强调电气安全准则，在涉及实际操作建议时必须添加安全警示语。

### 4.3.3 动态上下文注入实现
在实际执行过程中，`ExpertRegistry` 会根据 `TaskClassifier` 的路由结果，动态地将选定专家的 `system_prompt` 注入到请求的消息序列首位，确保模型在整个会话期间维持专家人格。

## 4.4 技能上下文注入实现

由于大模型的通用知识在处理极细分的电气标准（如特定的企业标准或最新行业规范）时存在局限，本系统实现了 `SkillRegistry` 机制。

### 4.4.1 技能定义格式
技能被定义为可复用的知识片段，存储在 `skills.yaml` 中。每个技能包含 ID、标题和具体的内容块。

```yaml
# skills.yaml 示例
SKILL_RELAY_PROTECTION_01:
  name: "距离保护定值计算标准"
  content: "在 220kV 线路中，距离保护第一段定值通常设置为线路长度的 80%-85%..."
SKILL_FAULT_ANALYSIS_02:
  name: "暂态稳定性判据"
  content: "等面积定值法要求加速面积小于减速面积，具体计算公式为..."
```

### 4.4.2 注入逻辑实现
系统通过 `get_context()` 方法实现 ID 列表到上下文字符串的映射。当某个专家被激活时，系统会检查该专家关联的 `tools` 或 `skills` 字段，将对应的知识片段拼接至提示词中。

```python
def get_context(self, skill_ids: list[str]) -> str:
    """将技能ID列表转换为拼接后的上下文字符串"""
    contexts = []
    for sid in skill_ids:
        if sid in self.skills_db:
            skill = self.skills_db[sid]
            contexts.append(f"【{skill['name']}】：\n{skill['content']}")
    return "\n\n".join(contexts)
```

在 `executor.py` 中，注入点位于消息构建阶段：
```python
# 动态注入技能上下文
skill_ctx = skill_registry.get_context(expert.skills)
if skill_ctx:
    system_parts.append(f"请参考以下专业知识库执行任务：\n{skill_ctx}")
```

## 4.5 V5 质量门控闭环实现

为了解决 LLM 在处理复杂电气计算时可能出现的逻辑错误，本系统引入了 **V5 质量门控（Quality Gate）** 机制，构建了一个“执行-评审-修正”的闭环。

### 4.5.1 CollaborationLoop 执行流程
该流程采用 `Writer-Reviewer` 对抗模型架构：
1.  **Writer（执行者）**：根据任务要求生成初步的分析报告或计算结果。
2.  **Reviewer（评审者）**：另一个独立专家智能体，负责根据专业标准对 Writer 的输出进行审核。
3.  **反馈环**：Reviewer 给出具体的修改意见，Writer 根据意见进行迭代。

### 4.5.2 评分函数与迭代条件
Reviewer 不仅提供文字建议，还必须输出一个量化的评分 $S$。评分由三个维度加权组成：
$$S = w_1 \cdot \text{Accuracy} + w_2 \cdot \text{Completeness} + w_3 \cdot \text{Compliance}$$
其中，$\text{Accuracy}$ 为计算准确度，$\text{Completeness}$ 为覆盖维度，$\text{Compliance}$ 为规范符合度。

当 $S < 85$ 分时，系统自动触发重新生成流程。为了防止死循环，设定最大迭代次数 $N_{max} = 3$。若达到最大次数仍不达标，系统将触发“辩论协议”，由一个更高阶的 `Supervisor` 模型进行最终裁决。

## 4.6 成本感知与多后端 Fallback

在工业级应用中，API 的可用性与成本是核心考量。本系统实现了 `MultiLLMClient` 的智能路由与容错机制。

### 4.6.1 多后端 Fallback 链
系统构建了一个优先级链条：`Primary` $\rightarrow$ `Fallback` $\rightarrow$ `Last Resort`。
- **Primary**：性能最强模型（如 GPT-4o 或 Claude 3.5），用于高复杂度逻辑推理。
- **Fallback**：性价比最高模型（如 DeepSeek-V3 或 Qwen-Max），用于常规任务。
- **Last Resort**：本地部署模型（如 Llama-3-8B），确保在公网中断时系统仍能提供基础服务。

### 4.6.2 智能探测与成本管理
`SmartInit` 模块在系统启动时会对各后端进行心跳探测，并缓存响应时间。在执行任务前，系统会根据任务复杂度预估 Token 消耗，若超过预算阈值，将自动将部分非核心步骤下放到轻量级模型。

```python
class MultiLLMClient:
    async def request_with_fallback(self, prompt, priority_list):
        for model_cfg in priority_list:
            try:
                response = await self.call_api(model_cfg, prompt)
                if self.validate(response):
                    return response
            except APIError as e:
                logging.warning(f"Model {model_cfg.name} failed: {e}")
                continue
        raise SystemError("All backend models are unavailable.")
```
